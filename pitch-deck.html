The solution

Three primitives stacked.
None depend on trusting the platform.

01 · Escrow

Permissionless auto-release

After the buyer's review window expires, anyone can call the on-chain ix to release payment. Silence becomes consent. The platform doesn't need to be online — the ix runs forever.

02 · Bids

Sealed in one place

One marketplace, not five scattered channels. Bid contents cryptographically sealed even from the buyer until bid_close_at. No favored vendor undercuts an early bid because the buyer cannot peek.

03 · Reputation

Anonymous · claim-based

Both sides stay anonymous through the lifecycle. Reputation merges into your public wallet only when you explicitly claim it. Merit accrues to the wallet that did the work.

Architecture

One product. Three layers, composed.

Browser

Client-side crypto + signing

  • ECIES key derivation (X25519 + XChaCha20-Poly1305)
  • HD keychain (HKDF, single SIWS sign per session)
  • Codama-generated TS client + @solana/kit signer
  • SNS resolver — every wallet renders as <handle>.tendr.sol

Cloak + MagicBlock PER

  • Cloak shielded pool · Groth16 ZK funds buyer + bidder ephemerals

Why Solana

Remove the chain. Every primitive breaks.

Without blockchain

  • Escrow held by a platform — platform shuts down, freezes funds, changes rules
  • Bids on a server — buyer admin can peek, vendor admin can leak
  • Reputation locked per platform — start over each time
  • Cross-border payouts via banks — 3-day settlement, KYC purgatory

With Solana

  • Escrow as on-chain ix — runs forever, permissionless auto-release
  • Sealed bids in TEE-gated PER — buyer cannot peek before reveal, by infrastructure not policy
  • Reputation as on-chain PDA — portable across products, claim-based
  • USDC settlement — minutes, no intermediaries, cross-border by default
Speaker note
This is the blockchain-necessity test. If a judge thinks "couldn't you just use Stripe + Hellosign?", this slide ends the thought. The TEE-gated peek-prevention is the sharpest claim — buyer literally cannot read the bid before close, because the on-chain ix that grants read access reverts. Not policy, not promise — code.
Frontier side tracks

Four side tracks.
Each one fails if its SDK is removed.

Cloak Track

Privacy precondition

Cloak's bar: "users would walk away the moment amounts and counterparties went public." Sealed-bid procurement is that product — a buyer who can be doxxed by Solscan won't post the RFP. We use private transfers + stealth-style HD-derived ephemerals + viewing-key-style claim attestation across attest_buyer_history / attest_win. Going deep on the core, not wide on demos.

MagicBlock · Privacy Track

Private DeFi · Auctions

Track explicitly names "Private DeFi — auctions, lending, trading primitives". Tender is a sealed-bid auction with full lifecycle. Bid commitments live inside Private ER; the reveal ix reverts before bid_close_at — sealed by infrastructure, not policy. Architecture and smart-contract design read across 28 ixs and 22 LiteSVM tests.

SNS Identity Track

On-chain reputation

Track lists "on-chain reputation systems" as the first example. Tender renders .tendr.sol across profiles, leaderboard, RFP cards, OG share cards. SNS is the identity layer that makes claim-based reputation legible — anonymous through the lifecycle, named when claimed. Sybil-resistant by design: rep is earned by completed work, not by self-attestation.

Tether · QVAC

Private AI

QVAC powers RFP scope drafting + structured bid drafting + post-decrypt comparison. Open-weight Qwen3 4B Q4_K_M on a dedicated Nosana GPU — no closed AI provider in the pipeline. App servers never see prompts or bid plaintexts. Roadmap: bundled QVAC for the agent-bid path, where local-first is the user.

What's not solved yet

V1 is trust-minimized by design.
V2 adds intelligence.

Disputes · v1 · live

No human moderator. On-chain escape hatches.

Both sides propose splits in a window. Default 50/50 after cool-off. Reputation gains and hits are real and on-chain — abuse is priced into the rep PDA the next time you bid. Trust-minimized, not trust-free. The starting point.

Disputes · v2 · planned

Bid-time agreed resolution.

  • Buyer + provider pick a resolution path before commit
  • QVAC AI-judge over the success criteria already on chain
  • Optional trusted-party audit · mix and match per RFP

Escrow lock model

Today: full lock at award. v2: optional milestone-by-milestone lock for buyers who want capital efficiency.

Mainnet

Post Frontier + audit review. Lean on already-deployed sponsor programs — no own mainnet program risk.

Agent mode

QVAC bundled local-first when agents bid for agents. AI agents don't need our hosted sidecar.

Speaker note
Sharpest judges will ask about disputes. Address it head-on. Brand-tone: don't oversell — call v1 trust-minimized, not trust-free. Name the gap, show the plan. The 50/50 default is a known surface for abuse, but reputation is the on-chain incentive that prices it. v2 turns the success criteria already required at bid-time into the input for AI/audit adjudication.
